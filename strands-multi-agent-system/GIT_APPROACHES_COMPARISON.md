# Git Approaches for Config-Only Branches - Complete Analysis

## üéØ **The Challenge**

Create a Git branch containing **ONLY** config files (~54 files), excluding all source code (~3,500+ files).

**The Problem**: Sparse-checkout filters the **working directory** but the **Git index** still references all files!

### ‚úÖ **Approach 1: `git add --sparse` (BEST - Git Native)**
### ‚úÖ **Approach 2: `git rm --cached` (CURRENT - Works but workaround)**
### ‚úÖ **Approach 3: Filter-Branch / Filter-Repo (Overkill for our use case)**
### ‚úÖ **Approach 4: Low-level Tree Building with `git mktree` (Most control)**

## ü•á **Approach 1: `git add --sparse` (RECOMMENDED)**

### **What It Is**

Git's **native solution** for sparse-checkout scenarios. The `--sparse` flag tells `git add` to respect the sparse-checkout configuration.

### **How It Works**

```python
# Enable sparse-checkout with cone mode DISABLED
repo.config_writer().set_value('core', 'sparseCheckout', 'true')
repo.config_writer().set_value('core', 'sparseCheckoutCone', 'false')

# Write sparse-checkout patterns
with open('.git/info/sparse-checkout', 'w') as f:
    for pattern in config_paths:
        f.write(f"{pattern}\n")

# Fetch and checkout (sparse-checkout filters working dir)
origin.fetch(main_branch, depth=1)
repo.git.checkout(f'origin/{main_branch}')

# Create orphan branch
repo.git.checkout('--orphan', new_branch_name)

# CRITICAL: Use git add --sparse
repo.git.add('--sparse', '.')  # ‚Üê The magic!

# Commit and push
repo.git.commit('-m', 'Config-only snapshot')
repo.git.push('origin', new_branch_name)
```

### **Why It's Better**

- ‚úÖ **Git-native solution** - designed for this exact use case
- ‚úÖ **Cleaner** - no manual index manipulation
- ‚úÖ **Respects sparse-checkout** automatically
- ‚úÖ **Less error-prone** - Git handles the filtering
- ‚úÖ **Future-proof** - official Git feature

### **Potential Issues**

- ‚ö†Ô∏è **Git version**: Requires Git 2.25+ (widely available)
- ‚ö†Ô∏è **Cone mode**: Must be disabled (`sparseCheckoutCone = false`) for wildcard patterns

### **When to Use**

- ‚úÖ When you have sparse-checkout configured
- ‚úÖ When you want the Git-native approach
- ‚úÖ For production systems (most reliable)

## üîß **Approach 2: `git rm --cached` (CURRENT)**

### **What It Is**

Manually clear the Git index, then add only files from working directory.

### **How It Works**

```python
# Sparse-checkout filters working dir to 54 files
repo.git.checkout(f'origin/{main_branch}')

# Create orphan branch (index still has 3,500 files)
repo.git.checkout('--orphan', new_branch_name)

# Clear the index completely
repo.git.rm('-rf', '--cached', '.')  # Index now EMPTY

# Add only from working directory
repo.git.add('.')  # Adds only 54 files

# Commit and push
repo.git.commit('-m', 'Config-only snapshot')
repo.git.push('origin', new_branch_name)
```

### **Why It Works**

- ‚úÖ **Explicit control** - you manually clear and rebuild
- ‚úÖ **Works with any Git version** - uses basic commands
- ‚úÖ **Predictable** - clear sequence of operations
- ‚úÖ **Already implemented** - we know it works

### **Downsides**

- ‚ö†Ô∏è **Not Git-native** - it's a workaround
- ‚ö†Ô∏è **Two-step process** - clear, then add
- ‚ö†Ô∏è **Manual verification** - need to check staged files

### **When to Use**

- ‚úÖ When `git add --sparse` is not available
- ‚úÖ For backward compatibility with old Git versions
- ‚úÖ When you need explicit control over staging

## üîÑ **Approach 3: Filter-Branch / Filter-Repo**

### **What It Is**

Rewrite Git history to remove unwanted files from commits.

### **How It Works**

```python
# Create branch normally (with all files)
repo.git.checkout('-b', new_branch_name, f'origin/{main_branch}')

# Filter the branch to keep only config files
repo.git.filter_branch(
    '--tree-filter',
    'find . -type f ! -name "*.yml" ! -name "*.yaml" ... -delete',
    '--',
    'HEAD'
)

# Or use git-filter-repo (modern replacement)
import git_filter_repo
git_filter_repo.FilterRepo(
    args=args,
    path_changes=path_changes
)

# Push the filtered branch
repo.git.push('origin', new_branch_name)
```

### **Why It's Interesting**

- ‚úÖ **Powerful** - can rewrite entire history
- ‚úÖ **Flexible** - can apply complex filters
- ‚úÖ **Preserves history** - can maintain commit chain

### **Downsides**

- ‚ùå **MASSIVE OVERKILL** for our use case
- ‚ùå **Slow** - rewrites entire commit history
- ‚ùå **Complex** - requires external tool (git-filter-repo)
- ‚ùå **Risk** - can corrupt repository if misused
- ‚ùå **Not needed** - we only need one commit!

### **When to Use**

- ‚ùå **DO NOT USE** for creating config-only branches
- ‚úÖ Use only when you need to rewrite existing history
- ‚úÖ Use for repository cleanup (remove secrets, large files)
- ‚úÖ Use for splitting repositories

## üõ†Ô∏è **Approach 4: Low-Level Tree Building (`git mktree`)**

### **What It Is**

Use Git's plumbing commands to manually build a tree object with only config files.

### **How It Works**

```python
# Step 1: Clone with sparse-checkout (54 files in working dir)
repo.git.checkout(f'origin/{main_branch}')

# Step 2: Build tree object manually
tree_entries = []
for file_path in checked_out_files:
    # Read file content
    with open(os.path.join(temp_dir, file_path), 'rb') as f:
        content = f.read()
    
    # Create blob object
    blob_sha = repo.git.hash_object('-w', '--stdin', input=content)
    
    # Add to tree
    tree_entries.append(f"100644 blob {blob_sha}\t{file_path}")

# Create tree object
tree_input = '\n'.join(tree_entries)
tree_sha = repo.git.mktree(input=tree_input)

# Step 3: Create commit pointing to this tree
commit_sha = repo.git.commit_tree(
    tree_sha,
    '-m', 'Config-only snapshot'
)

# Step 4: Create branch pointing to this commit
repo.git.update_ref(f'refs/heads/{new_branch_name}', commit_sha)

# Step 5: Push the branch
repo.git.push('origin', new_branch_name)
```

### **Why It's Interesting**

- ‚úÖ **Maximum control** - you build the tree manually
- ‚úÖ **Explicit** - you decide exactly what goes in
- ‚úÖ **Educational** - understand Git internals
- ‚úÖ **Precise** - no surprises

### **Downsides**

- ‚ö†Ô∏è **Complex** - requires understanding Git internals
- ‚ö†Ô∏è **Verbose** - lots of code
- ‚ö†Ô∏è **Manual work** - you handle everything
- ‚ö†Ô∏è **Error-prone** - easy to make mistakes
- ‚ö†Ô∏è **Not needed** - high-level commands work fine

### **When to Use**

- ‚úÖ When you need **absolute control** over tree structure
- ‚úÖ When you're building **synthetic commits** (testing, tooling)
- ‚úÖ When other approaches fail
- ‚úÖ For **learning Git internals**

## üéì **Key Learnings**

### **Why Sparse-Checkout Alone Doesn't Work**

```
Sparse-Checkout filters: Working Directory ‚úÖ
Sparse-Checkout does NOT filter: Git Index ‚ùå

Result:
  Working Dir: 54 files ‚úÖ
  Git Index: 3,500 files ‚ùå (still references all files from origin/main)
  Commit: Uses Git Index ‚Üí 3,500 files ‚ùå
```

### **Why We Need Extra Steps**

```
git checkout --orphan new_branch
  ‚Üí Creates new branch
  ‚Üí Copies working directory ‚úÖ
  ‚Üí Also copies Git index ‚ùå (includes all files)

Solution 1 (Git-native):
  git add --sparse .
  ‚Üí Adds only files matching sparse-checkout

Solution 2 (Manual):
  git rm --cached .  ‚Üí Clear index
  git add .          ‚Üí Add only from working dir
```
